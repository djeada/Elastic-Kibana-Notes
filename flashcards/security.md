### 1. **What are the layers of Elasticsearch security?**

Elasticsearch security is organized into authentication (verifying user identity through realms), authorization (controlling access via role-based access control), encryption (TLS for transport and HTTP layers), audit logging (recording security events), and IP filtering (restricting network access by IP address), all of which are enabled by default in Elasticsearch 8.x.

### 2. **What changed in Elasticsearch 8.0 regarding security?**

Starting with Elasticsearch 8.0, security features including TLS encryption, authentication, and RBAC are enabled by default out of the box; the setup process auto-generates CA and node certificates, sets the elastic user password, and creates an enrollment token for adding new nodes and connecting Kibana, eliminating the previous need for manual security configuration.

### 3. **What is a realm in Elasticsearch authentication?**

A realm is an authentication backend that verifies user credentials; Elasticsearch supports multiple realm types including native (stored in a system index), file (users defined in a local file), LDAP and Active Directory (external directory services), SAML and OIDC (single sign-on via identity providers), PKI (client certificate), and API key, evaluated in a configurable order.

### 4. **What are the built-in users in Elasticsearch?**

Built-in users include elastic (superuser for administration), kibana_system (used by Kibana to communicate with Elasticsearch), logstash_system (for Logstash monitoring), beats_system (for Beats monitoring), apm_system (for APM server), and remote_monitoring_user (for cross-cluster monitoring); their passwords should be set immediately after installation using the elasticsearch-setup-passwords tool or the change password API.

### 5. **What is role-based access control (RBAC)?**

RBAC assigns users one or more roles, each defining a set of cluster privileges (e.g., manage, monitor) and index privileges (e.g., read, write, create_index) on specific index patterns, optionally restricted by field-level and document-level security; users inherit the union of all privileges from all their assigned roles.

### 6. **What are cluster privileges?**

Cluster privileges control cluster-wide operations such as manage (full cluster administration), monitor (read-only cluster health and stats), manage_security (user and role management), manage_index_templates, manage_pipeline, and manage_ilm; they are granted to roles and determine what administrative actions a user can perform.

### 7. **What are index privileges?**

Index privileges control what operations a role can perform on specific indices, including read (search and get), write (index and delete documents), create_index, delete_index, manage (index settings and mappings), and monitor (index stats); they are scoped to index name patterns so a single role can grant different access to different indices.

### 8. **What is field-level security (FLS)?**

Field-level security restricts which fields within a document a role can see, so that when a user with FLS restrictions runs a search they only receive the allowed fields in the _source and cannot filter or aggregate on restricted fields, enabling fine-grained data access within a shared index.

### 9. **What is document-level security (DLS)?**

Document-level security applies a query filter to a role so that users with that role can only see documents matching the filter, effectively creating a per-role view of the index; this is useful for multi-tenant deployments where different users should only access their own data within the same index.

### 10. **How do you create a custom role?**

Use the PUT _security/role API or the Kibana Stack Management UI to define a role with a name, optional cluster privileges, one or more index privilege entries (specifying index patterns, privileges, and optional FLS/DLS), and optional Kibana space privileges; then assign the role to users or API keys.

### 11. **What are API keys and how are they used?**

API keys are credentials generated by Elasticsearch that grant specific privileges without requiring a username and password, useful for programmatic access from applications, scripts, or CI/CD pipelines; they are created via the _security/api_key API, can have an expiration time, and are sent in the Authorization header as an ApiKey base64 token.

### 12. **What are service tokens?**

Service tokens are special-purpose credentials for Elastic Stack components (primarily Kibana and Fleet) that authenticate service accounts without using shared user credentials; they are created with the elasticsearch-service-tokens CLI tool and stored securely on the node, providing a more secure alternative to configuring username/password in configuration files.

### 13. **How does TLS work in Elasticsearch?**

TLS encrypts communication on the transport layer (node-to-node) and the HTTP layer (client-to-node); Elasticsearch requires transport TLS for cluster formation, and the elasticsearch-certutil tool generates a Certificate Authority and signed certificates, which are configured in elasticsearch.yml via xpack.security.transport.ssl and xpack.security.http.ssl settings.

### 14. **What TLS verification modes are available?**

The verification_mode setting controls how Elasticsearch validates peer certificates: full (default) verifies the certificate chain and hostname, certificate verifies only the chain without hostname checking, and none disables verification entirely; production environments should always use full mode for both transport and HTTP layers.

### 15. **What is audit logging and what events should it track?**

Audit logging records security-relevant events to a file or index, including authentication successes and failures, access denied events, role changes, and API key creation or invalidation; it is essential for compliance and forensic analysis, and can be configured to include or exclude specific event types and users to manage log volume.
